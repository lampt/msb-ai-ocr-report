1. Phân loại năng lực OCR và Mức độ trưởng thành 

Năng lực OCR được hệ thống hóa thành 5 cấp độ tiến hóa, đi kèm với các bộ chỉ số đo lường (Metric) và ý nghĩa nghiệp vụ cụ thể:

### 1.1. 

Bảng tiêu chuẩn đánh giá mức độ trưởng thành 

| Cấp độ | Công nghệ | Metric chính | Chuẩn benchmark | Ý nghĩa nghiệp vụ |
| --- | --- | --- | --- | --- |
| **Level 1** | Basic OCR | Character Accuracy Rate (CAR) | 85-90% (text rõ ràng) | Scan lưu trữ, tra cứu cơ bản.

 |
|  |  | Word Accuracy Rate (WAR) | 80-85% | Giảm nhập liệu tay với văn bản.

 |
|  |  | Processing Speed | 1-2 trang/giây | Khả năng xử lý khối lượng lớn.

 |
| **Level 2** | Template-based OCR | Field-level Accuracy | >90% trên form cố định | Trích xuất thông tin CCCD, điện/nước.

 |
|  |  | Form-level Success Rate | >85% | Đảm bảo hồ sơ eKYC đi thẳng.

 |
|  |  | Rejection Rate | <10% | Giảm workload nhập tay.

 |
| **Level 3** | AI-powered OCR/NLP | Field-level Accuracy (chữ tay + in) | 80-90% | Hợp đồng, đơn vay có chữ ký.

 |
|  |  | Document Classification Accuracy | >90% | Tự phân loại tài liệu (hợp đồng, sổ đỏ...).

 |
|  |  | Manual Intervention Rate | <5-8% | Tăng tự động hóa LOS/Bank.

 |
| **Level 4** | OCR + GenAI (Doc Understanding) | Information Extraction F1 | >0.80 | Trích xuất điều khoản, số liệu.

 |
|  |  | QA Accuracy on Docs | >75% | Chatbot QA trên hợp đồng.

 |
|  |  | Summarization Quality (ROUGE-L) | 0.5-0.6 | Tóm tắt hợp đồng tín dụng.

 |
| **Level 5** | OCR + VLM (Agentic OCR) | Multi-hop QA Accuracy | >70% | Trả lời câu hỏi suy luận phức tạp.

 |
|  |  | Hallucination Rate | <5% | Giảm rủi ro pháp lý khi AI tự chế.

 |
|  |  | Automation Rate | >80% E2E | Hồ sơ vay được xử lý hoàn toàn tự động.

 |

2. Đánh giá hiện trạng ứng dụng OCR tại MSB 

### 2.1. 

Thực trạng nghiệp vụ và vận hành 

* 
**Phụ thuộc xử lý thủ công:** Nhiều bước nhập liệu, đối chiếu và kiểm tra chứng từ bằng mắt thường.


* 
**Quy trình dựa trên giấy tờ:** Còn phân tán, thiếu tính tự động hóa cao.


* 
**Rủi ro sai sót:** Dễ lỗi nhập liệu, bỏ sót thông tin; kết quả phụ thuộc cá nhân, khó chuẩn hóa.


* 
**Hiệu suất:** Thời gian xử lý kéo dài, khó đáp ứng SLA khi khối lượng giao dịch tăng.


* 
**Kiểm soát:** Thiếu công cụ tự động theo dõi tiến độ và truy vết rủi ro vận hành.


* 
**Hệ thống ảnh hưởng:** BPM Ops, BPM Risk, Digi-lending, IB Corp, IB Admin... đều tồn tại nhiều bước thủ công.



### 2.2. 

Chất lượng dữ liệu đầu vào 

* 
**Đa dạng nguồn:** Khách hàng, đối tác và nội bộ cung cấp qua scan, chụp ảnh, upload file.


* 
**Định dạng không chuẩn hóa:** Bao gồm PDF, Word, ảnh (JPG, PNG), bản photocopy với độ phân giải và chất lượng không đồng nhất.


* 
**Chất lượng thấp:** Hình ảnh thường bị mờ, nghiêng, lệch góc, thiếu sáng hoặc lóa sáng.


* 
**Nhiễu dữ liệu:** Nhiễu nền, bóng tay, dấu gấp, vết mực và đặc biệt là tình trạng đóng dấu chồng chữ.


* 
**Nội dung phức tạp:** Văn bản đa ngôn ngữ (Việt có dấu, Anh, Trung), ký hiệu nghiệp vụ và bố cục bảng biểu nhiều cột/dòng.


* 
**Tính đặc thù pháp lý:** Chứa watermark, dấu đỏ, chữ ký tay và các mã số dài.

5. Phân tích lỗi và kết luận kỹ thuật 

### 5.1. 

Nhược điểm của giải pháp In-House (AIEP) 

* 
**Ngôn ngữ:** Hay mất/lẫn dấu tiếng Việt ở các ký tự như â, ã, ơ, ô.


* 
**Thành phần đặc biệt:** Chưa nhận diện được chữ ký tay, con dấu và các vùng ảnh quan trọng.


* 
**Cấu trúc:** Gặp khó khăn lớn với bảng biểu phức tạp và chữ viết tay.


* 
**Vận hành:** Hay bị lỗi OOO (Out of Memory) hoặc hết tài nguyên CUDA.


* 
**Đầu ra:** Chỉ OCR ra text thuần, chưa có cấu trúc thông minh.



### 5.2. 

Nhược điểm của giải pháp VNG Cloud 

* 
**Nhầm lẫn ký tự:** Nhầm Y-V, M-N, nhầm dấu "=" thành "-", "|" thành "1".


* 
**Lỗi hậu xử lý:** Mô hình sửa lỗi chính tả đôi khi tự ý thêm thông tin không liên quan hoặc sai dấu (sỉ thành sĩ).


* 
**Dữ liệu số:** Nhầm số điện thoại (ví dụ: 8989 thành 8998).


* 
**Độ ổn định:** Chất lượng giảm mạnh trên các bản in/scan chất lượng thấp hoặc chữ viết tay xấu.


* 
**Vận hành:** API không "always on", không thể đáp ứng vai trò IT Cap của MSB.

6. Khuyến nghị chiến lược giải pháp OCR 

### 6.1. 

Phân bổ Engine theo loại hồ sơ 

| Giải pháp | Nhóm giấy tờ áp dụng |
| --- | --- |
| **In-House (AIEP)** | Điều lệ doanh nghiệp, Quy chế tài chính/nội bộ, Thỏa thuận cấp bảo lãnh, HĐLĐ/Quyết định làm việc.

 |
| **VNG Cloud** | Sổ sách chứng từ nhập/bán hàng, Hóa đơn, Báo cáo tài chính, Văn bản hành chính (Quyết định/Thông báo), Khế ước nhận nợ.

 |
| **Finetune In-House** | CCCD, Giấy tờ nhà đất, BB đánh giá tài sản (vì chứa thông tin nhạy cảm PII và theo template đặc thù).

 |

### 6.2. 

Năng lực đề xuất phát triển trên AIEP 

* 
**OCR Routing & Model Selection:** Tự động lựa chọn Engine phù hợp dựa trên loại tài liệu, độ phức tạp và yêu cầu độ chính xác/tốc độ.


* 
**Cơ chế Fallback:** Cho phép chuyển sang engine khác khi engine ưu tiên gặp lỗi, giảm rủi ro phụ thuộc vendor.


* 
**Monitoring & Observability:** Xây dựng Dashboard realtime theo dõi số lượng request, tỷ lệ thành công/lỗi, thời gian xử lý và cảnh báo sớm lỗi model/giảm chất lượng.


* 
**Tối ưu hóa model:** Finetune lại model In-house để cải thiện khả năng xử lý tiếng Việt và hiệu suất hệ thống.
